# Murmur Codebase Audit — 2026-07-11

Multi-agent audit across 9 dimensions (data durability, database, audio pipeline, Rust quality, security/privacy, frontend, LLM/summary, testing/CI, product UX). Every critical/high finding was independently re-verified against the actual code by an adversarial second agent; findings the verifier refuted are excluded. File references are `path:line` at the time of the audit.

Context that shaped severity: this app is the owner's personal note taker for all jobs. A lost recording, lost transcript, or corrupted database is the worst possible failure.

---

## P0 — Silent data-loss paths (fix first)

### 1. The meeting save is all-or-nothing, webview-driven, and silently skippable
- Rust deliberately skips the DB save at stop: `recording_commands.rs:761` ("Database save removed - frontend will handle this"). The only durable write is `useRecordingStop.ts:230`, gated on `if (isCallApi && transcriptionComplete == true)`, built from React in-memory state (`transcriptsRef`).
- The gate is broken in both directions: the registered `get_transcription_status` is a stub (`lib.rs:218-224` hardcodes `is_processing: false`), and a single status-poll error exits the wait loop with `transcriptionComplete = false` → the else branch just does `setStatus(IDLE)` — **no save, no toast, no error** (`useRecordingStop.ts:324-327`).
- Any `transcription-error` / `transcript-error` event mid-meeting calls `onRecordingStop(false)` (`RecordingControls.tsx:251-300`) — the no-save stop flow — while Rust keeps recording. One transient chunk error can cost the whole meeting.
- Segments transcribed after Stop is pressed are also excluded from the on-disk `transcripts.json` (listener unlistened at `recording_commands.rs:537` before the up-to-10-minute backlog drain).
- Tray Quit is `app.exit(0)` with no `IS_RECORDING` check (`tray.rs:50`); `RunEvent::Exit` only checkpoints the DB — quitting mid-recording or during the ~4.5s post-stop window loses the meeting from SQLite.

**Fixes** (in increasing ambition):
1. Always save when `isCallApi` is true, regardless of `transcriptionComplete`; on timeout save partial + warning toast. (small, do immediately)
2. Treat transcription errors as toasts, never as stop triggers. (small)
3. Gate tray Quit on `IS_RECORDING` (confirm or auto stop-and-save). (small)
4. **Structural fix: make Rust the writer of record.** Create the meeting row at record-start and INSERT segments into SQLite as they are produced (the `transcript-update` listener in `recording_commands.rs` already sees every segment). Kills the entire class: webview reloads, dropped events, timeouts, quit races all become non-fatal. (large — the single highest-value change in this audit)

### 2. Recording can silently stop persisting audio mid-meeting
- `encode.rs:74/83`: `command.spawn().expect(...)` / `wait_with_output().unwrap()` — an ffmpeg failure (AV quarantine, missing binary) panics the accumulation task; its JoinHandle is dropped, and the pipeline keeps feeding a dead channel via `let _ = sender.send(...)` (`pipeline.rs:853`). UI shows recording; nothing is being saved.
- `incremental_saver.rs:78-81`: on checkpoint failure the buffer is never cleared → every ~600ms chunk retries a full re-encode of an ever-growing buffer; disk-full becomes an unbounded-RAM spiral (~1.4 GB/hour) ending in OOM and total loss.
- `recording_saver.rs:160-163`: meeting-folder init failure logs "Continue anyway - will use fallback flat structure" — **no fallback exists**. The whole meeting records with zero audio checkpoints and zero transcripts.json, and stop reports success.
- Disk-full / write errors anywhere in this path are log-only (`recording_saver.rs:205-207`); no event reaches the UI.

**Fixes**: replace expect/unwrap with Result propagation; emit the existing `recording-error` event on add_chunk/checkpoint failures and show a persistent banner; abort start_recording if folder init fails; check free disk space at start; consider pure-Rust WAV checkpoints (hound) so persistence never depends on spawning ffmpeg every 30s; keep and monitor the accumulation JoinHandle (watchdog).

### 3. Device failure handling is dead code end-to-end — dead mic records silence with no warning
- `poll_audio_device_events` (`recording_commands.rs:1008`) is documented "should be called every 1-2s by frontend" — **zero frontend callers**. The `recording-error` event emitted at `recording_commands.rs:229-231` has **zero frontend listeners**. `handle_device_disconnect`/`handle_device_reconnect` (`recording_manager.rs:562/581`) are never called by anything.
- When a Bluetooth/USB device dies, the ring buffer zero-pads the dead channel (`pipeline.rs:115-118`) and the recording continues as silence indefinitely.
- `report_error`'s internal stop (`recording_state.rs:297-311`, after 10 recoverable errors or 1 fatal) only flips atomics — streams keep running, saver never finalizes, `IS_RECORDING` global and the UI still say recording.
- Windows: loopback is pinned to the endpoint resolved at start with fuzzy `name.contains()` matching and silent default fallback (`devices/platform/windows.rs:118/172-252`). Docking or Windows switching the default playback device kills system-audio capture silently. System-stream creation failure at start is warn-and-continue (`stream.rs:406-416`) — mic-only recording with no signal.
- The recording waveform is fake: `Math.random()` bars on a 300ms interval (`page.tsx:169-179`). Real `audio-levels` events exist (used only in the settings device picker) and the `speech-detected` state is set but never rendered (`RecordingControls.tsx:59,306-309`).

**Fixes**: push DeviceEvents as Tauri events + persistent banner + auto-reconnect with backoff; add a frontend listener for `recording-error`; route error-stops through the full stop path; drive the bars from real mic/system RMS; add a silence watchdog ("no speech detected for 5 min"); poll the default output endpoint and re-bind loopback on change; emit `system-audio-unavailable` when loopback setup fails.

### 4. The database has no safety net
- **WAL deletion "recovery"** (`database/manager.rs:81-97`): any startup error whose message contains "malformed"/"corrupt" deletes the `-wal`/`-shm` files with **no backup**. After a crash (the exact case where it triggers), the WAL holds the newest committed meetings — they're destroyed silently. Confirmed independently by 5 reviewers.
- **No backup mechanism anywhere** and a failed migration **panics the app at startup** (`lib.rs:503-506` `.expect("Failed to initialize database")`) — a bad migration locks you out of your entire history with no UI.
- **Legacy import can truncate the source DB to zero bytes**: `manager.rs:147-154` does `fs::copy(legacy, target)` where onboarding passes `legacy == target` (`commands.rs:87-104` + `OnboardingContext.tsx:199-202` auto-runs it on first launch). Same-path copy truncates on macOS/Linux; import also never copies `-wal` sidecars and silently no-ops (reporting success) when a DB already exists.
- Meeting deletion is a hard delete with one confirm, no trash/undo (`meeting.rs:278-318`).

**Fixes**: `VACUUM INTO` rotating backups (startup/daily, keep N) + snapshot before every migration run (small!); quarantine instead of delete in WAL recovery + user-visible notice; error window instead of panic on migration failure; same-path guard + `-wal` copy + merge-or-fail semantics in import; soft-delete or undo-toast for deletion.

### 5. Crash recovery has holes exactly where it's needed
- Recovery reads **only** webview IndexedDB (`useTranscriptRecovery.ts`); the on-disk `transcripts.json` that Rust writes atomically after every segment is **never read by any recovery path**.
- `page.tsx:82` runs `deleteOldMeetings(7)` before checking for recoverable transcripts, and it deletes without checking `savedToSQLite === false` — unsaved crash data is purged after 7 days (and the recovery hook self-filters to 7 days too).
- After a webview reload mid-recording, `currentMeetingId` is lost, so IndexedDB journaling silently stops (`TranscriptContext.tsx:325-327`).
- Recovered segments drop the `speaker` field (`useTranscriptRecovery.ts:163-174`) — attribution permanently lost for exactly the meetings that crashed.
- Checkpoint recovery sorts filenames lexicographically while names use `{:03}` — recordings past ~8.3h reassemble out of order (`incremental_saver.rs:293` vs `:102`).
- Audio finalization failure (ffmpeg missing) is swallowed (`recording_manager.rs:305-316`): meeting saved, audio silently absent, checkpoints orphaned with no UI to merge them.

**Fixes**: Rust-side startup scan for meeting folders with `metadata.json` status `"recording"` → offer import of transcripts.json + checkpoint merge (independent of IndexedDB); never purge unsaved IndexedDB entries; restore `currentMeetingId` from sessionStorage after reload; include `speaker` in recovery mapping; numeric sort for checkpoints; surface finalization failure + "merge checkpoint audio" action.

## P1 — Silently wrong output

### 6. Summaries can silently cover a fraction of the meeting
- **Ollama truncation** (`summary/service.rs:415-433`, `ollama/metadata.rs:225-248`): chunking threshold uses the model's *max trained* context (e.g. 131072) but requests go through the OpenAI-compat endpoint with **no `num_ctx`** — Ollama serves its default (~4k) and silently truncates. A 2-hour meeting yields a "successful" summary of the first few minutes. Fix: native `/api/chat` with `options.num_ctx`, verify `prompt_eval_count`.
- **Claude hard-capped at 2048 output tokens** (`llm_client.rs:268`), user's max_tokens setting only applies to CustomOpenAI, and `stop_reason` is never checked — long summaries and the Spanish translation pass truncate mid-sentence and save as "completed".
- **Failed chunks silently dropped** (`processor.rs:474-493`): one transient error on chunk 4/10 removes ~20 min of the meeting with a success toast. No retry exists anywhere in the pipeline (`llm_client.rs` sends exactly one request per call).
- Translation failure discards the completed English summary (`processor.rs:608`).
- Chat truncates the transcript to the **first 30k chars** with no UI indication (`chat_api.rs:11,273-277`) — "what did we decide at the end?" fails on long meetings.
- Summary processes interrupted by quit stay PENDING forever (no startup sweep); frontend poller times out at ~16.5 min while backend legitimately runs longer, inviting concurrent duplicate generations that corrupt the cancellation registry.
- Cloud providers: no chunking, no 429/5xx retry, raw JSON body as user-facing error; sidecar stdin/stdout has no request IDs (concurrent chat + summary can swap responses).

### 7. Search is broken for Spanish
- `transcript.rs:446-467` `get_match_context` byte-slices the original string at ±100 using offsets from the lowercased copy — **panics** whenever a multi-byte char (á, é, ñ…) straddles the window. Search appears permanently broken; the sidebar shows a stuck "Searching…". The UTF-8-safe `snippet()` in `mcp/tools.rs:153` already solves this — port or share it.
- `LOWER(t.transcript) LIKE ?` is ASCII-only folding (accented uppercase unfindable), unindexed full-scan, no LIMIT, fired per keystroke with no debounce and no response sequencing; covers transcripts only (not summaries/notes/chat/legacy chunks).
- Long-term: FTS5 with `unicode61 remove_diacritics 2` across transcripts + summaries + chat; also reusable by the MCP search tool.

### 8. Live transcription quality
- The live path downsamples 48k→16k with a ~2-tap moving average + linear interpolation (`vad.rs:112-160`) — the method the project itself abandoned in `decoder.rs` for aliasing — and Whisper transcribes *those* samples. Imported files get the good rubato sinc path; live meetings get the bad one. Reuse the existing `SincFixedIn` machinery.
- Up to 600ms of tail audio is discarded on every stop (`pipeline.rs:876-919` flushes VADs but never drains the ring buffer); window comments say 50ms but code is 600ms.
- System audio gets no loudness normalization before VAD (`pipeline.rs:262-266`) while mic gets EBU R128 — quiet remote participants can be VAD-gated out of the transcript entirely.
- Zero-padding of late streams ratchets mic/system desync over long meetings (`pipeline.rs:104-134`).

## P2 — Infrastructure & correctness debt

### 9. No CI at all
- All 8 workflows are `workflow_dispatch`/`workflow_call` only. `pr-main-check.yml` greps a version string and never fails. **Nothing gates main.**
- **229 existing Rust tests are never executed by any workflow.** Release signs (DigiCert/notarization) and publishes auto-update artifacts straight from main with no test gate — the v0.5.0 crash-on-transcribe class ships again unimpeded.
- Zero tests on: all DB repositories (including destructive `merge_segments`/`split_segment`), all 15 migrations, `pipeline.rs` mixing math, `recording_saver.rs`, `recording_manager.rs`, and the crash-recovery merge path (2 trivial tests in `incremental_saver.rs`).
- `release.yml:59` can mint non-semver `0.6.0.1` versions that desync tag/manifest/binary and break the updater.
- Frontend tests exist but are unrunnable (bun:test imports, no runner, no `test` script).

**Highest-value additions**: `ci.yml` on push/PR (cargo test + clippy + tsc/eslint, required check); sqlx in-memory repo+migration tests; WAL-recovery preservation test; pipeline mixing property tests; incremental_saver crash-recovery tests; MCP-based smoke test of the release binary.

### 10. Rust crash-safety
- Four `unsafe impl Send` on cpal Stream wrappers (`stream.rs:29,38,363`, `recording_manager.rs:35`) with a stated invariant the code does not uphold — WASAPI COM streams created/dropped from arbitrary tokio threads; UB manifesting as crash-on-stop. Fix: dedicated audio-ownership thread with a command channel (also removes the reconnect lock-across-await and simplifies single-stream reconnect).
- `AudioPipeline::new` panics on VAD model load failure (`pipeline.rs:692,700`) → Record button hangs forever with mic claimed.
- Whisper/Parakeet inference and ffmpeg encode run inline on tokio workers (no `spawn_blocking`); all audio channels unbounded; the stop path abandons the backlog after 10 minutes.
- `transcripts.json` fully rewritten (pretty-printed) on every segment — O(n²) I/O; switch to append-only JSONL + fsync before rename (current temp+rename never calls `sync_all`).
- TOCTOU on `IS_RECORDING` allows double-start (`recording_commands.rs:84` vs `:247`); use `compare_exchange`.
- `read_audio_file` loads whole audio into memory in a sync command on the main thread (`lib.rs:227`).

### 11. Security & privacy (the app's brand promise)
- **All API keys plaintext in SQLite** (`setting.rs:101-110,201-210`): OpenAI/Anthropic/Groq/OpenRouter/Deepgram/ElevenLabs/pyannote/HF token. Move to OS keychain (keyring crate) with one-time migration.
- **No integrity verification on any download**: Parakeet v3 comes from `meetily.towardsgeneralintelligence.com` (upstream project's server — infrastructure this fork doesn't control), `uv` from GitHub `latest` (not even version-pinned), whisper/GGUF models unpinned. Executable + in-process ONNX parsing = realistic supply-chain RCE. Pin SHA-256 per artifact.
- `open_external_url` on Windows shells `cmd /C start <url>` (`api.rs:1280`) — metacharacter injection class; use tauri-plugin-opener + scheme allowlist.
- Webview capability grants `fs:read-all`/`fs:write-all` (tauri.conf.json) — any XSS/dep compromise gets the whole filesystem. Scope to app-data + recordings.
- Full transcript text logged at info level with `RUST_LOG=info` forced in `main.rs:53`; demote to `perf_debug!`.
- pyannote.ai cloud diarization uploads complete meeting audio (~115 MB/hour) — ensure explicit, visible "leaves this device" labeling; same for cloud LLM providers (a persistent local/cloud indicator would fit the brand).
- CSP still allowlists dead ports 5167/8178; `serverAddress` plumbing in SidebarProvider is vestigial.

### 12. Frontend state bugs (verified)
- `SidebarProvider.tsx:277-282` clears **all** summary polling intervals whenever the polls map changes — start a summary on meeting B and meeting A's poll dies ("Generating…" forever). Keep intervals in a ref / switch to Tauri events.
- Refetch-unmounts-PageContent confirmed still present (`meeting-details/page.tsx:354`); RediarizeDialog works around it, RetranscribeDialog doesn't. Render overlay instead of unmounting.
- Live view virtualization is structurally defeated (`TranscriptPanel.tsx:53` outer scroller + `h-full` against auto-height parent in `VirtualizedTranscriptView.tsx:440`) — thousands of DOM rows + full re-sort per batch exactly during long meetings.
- `useRecordingStateSync` is dead code: gated on `window.__TAURI__` which Tauri 2 never injects without `withGlobalTauri` (`useRecordingStateSync.ts:50`). Use `isTauri()`.
- Auto-summary can race config load and fire with hardcoded `ollama/llama3.2:latest` (`page-content.tsx:150`, `ConfigContext.tsx:100-105`).
- Recovery/reload gaps: after reload, `status`/polling not restored (`RecordingStateContext.tsx:223-226`).
- Native `alert()` on recording failures blocks the webview (`RecordingControls.tsx:80,214,232`; `useRecordingStart.ts:229,312`).

## Product gaps for the "career of meetings" use case

1. **Backup/restore + bulk export** — the only exit today is one meeting's markdown at a time (`export.rs`). Settings > Data: VACUUM INTO backup, restore, "export all to Obsidian-compatible folder", scheduled auto-backup. (Overlaps with P0 #4.)
2. **Organization**: sidebar is a flat, undated, title-only list (`SidebarProvider.tsx:113-122`; `api_get_meetings` strips `created_at`). Add date display + day/week grouping (cheap), then tags/workspaces per job with filter chips (one `meeting_tags` migration). Scope MCP tools by tag too.
3. **Recording trust HUD**: real level meters, speech-detected badge, silence watchdog, device banners (P0 #3 covers the plumbing).
4. **Global hotkey quick-record** (tauri-plugin-global-shortcut → Rust-side start, no webview navigation): today the panic path is tray-menu → page reload → model validation. Consider an opt-in 30-60s pre-roll ring buffer.
5. **Post-stop title prompt with AI-suggested name** — default `Meeting 11_07_26_09_00_01` titles are the root of unfindability.
6. **FTS5 unified search** (transcripts + summaries + notes + chat, diacritic-insensitive) + find-in-transcript with scroll-to-match; deep-link search results to the segment.
7. **Cross-meeting action items**: extract action-item sections into a table during summarization; "Open follow-ups" view filterable by tag/job; new MCP tool.
8. **Calendar-aware capture** (auto-name from concurrent event, "meeting starting — record?" notification).
9. Per-meeting ES/EN/Auto language quick-pick next to the record button.
10. Startup self-check before recording: ffmpeg runnable, folder writable, free space, both streams delivering within 2s.

## Suggested sequencing

**Week 1 (small diffs, huge risk reduction):**
1. Always-save on stop + never stop on transcription errors + tray-quit guard (P0 #1 items 1-3)
2. Rotating `VACUUM INTO` backups + snapshot-before-migration + quarantine-not-delete WAL recovery (P0 #4)
3. Surface recording-error / add_chunk failures + disk-space check; fix `encode.rs` expect/unwrap (P0 #2)
4. Fix search UTF-8 panic (port `mcp/tools.rs::snippet`) + debounce (P1 #7)
5. `ci.yml` on push/PR running the 229 existing tests + clippy + tsc (P2 #9)
6. Don't purge unsaved IndexedDB meetings; include `speaker` in recovery (P0 #5)

**Month 1 (structural):**
- Rust-side authoritative incremental persistence (meeting row at start, segments as they stream) — makes most of P0 #1/#5 obsolete
- Wire device events + real levels + silence watchdog end-to-end
- Ollama `num_ctx` + Claude max_tokens + chunk retry + partial-summary warnings
- Filesystem-based recovery scan (metadata.json status == "recording")
- Keychain secrets migration + SHA-256 pinning for downloads
- sqlx repo/migration tests + WAL-recovery test + pipeline math tests

**Quarter (product):**
- Tags/workspaces + dated sidebar; FTS5 search; backup/restore UI + bulk export
- Global hotkey; recording HUD; action-items view
- Dedicated audio thread (remove unsafe Send); JSONL transcript journal; streaming summaries

## Findings the adversarial pass refuted (excluded above)

- "RECORDING_MANAGER std-mutex held across .await starves the runtime" — the reconnect path holding it is currently unreachable dead code (nothing calls it); keep in mind if you wire reconnect up.
- "Stop with 'No recording in progress' leaves a permanent spinner" — backend guard makes the race practically unreachable.
- "Transcript listener re-registration gap drops segments at recording start" — temporally guarded in practice (first event arrives well after re-registration).
- "Transcripts only persisted to SQLite at stop" as a standalone DB finding — true, but downgraded because transcripts.json + IndexedDB journaling exist; the real risk is the save-gate logic (P0 #1).

*Audit run: 45 agents, ~6M tokens, 2 independent passes over most dimensions; every P0/P1 "confirmed" item was verified against source by a second agent instructed to refute it.*
